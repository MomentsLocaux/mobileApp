# Audit RLS — Moments Locaux (projet Supabase `prymkgkafaovhzopslea`)

Date : 2026-06-05 · Lecture seule + Lot 1 appliqué · Source de vérité : base prod.

## Méthode

`pg_class.relrowsecurity` (RLS on/off) croisé avec `pg_policies` pour chaque
table du schéma `public`. Rappel : **RLS désactivé = les policies sont inertes
et la table est totalement ouverte** aux rôles `anon` / `authenticated`.
Le scraper utilise `SUPABASE_SERVICE_ROLE_KEY` (bypass RLS).

## Lot 1 — APPLIQUÉ (migration `enable_rls_ingestion_tables`)

| Table | Avant | Après | Justification |
|---|---|---|---|
| `event_import_staging` | RLS off, 0 policy | RLS on | Aucune policy → verrouillée pour anon/auth. Scraper en service_role. Non lue par mobile/WebConsole. |
| `event_subcategory` | RLS off, SELECT public | RLS on | Lecture taxonomie conservée (policy SELECT anon+auth). Écritures anon/auth bloquées. |
| `event_tag` | RLS off, SELECT public | RLS on | Idem `event_subcategory`. |

Usages vérifiés : mobile (`taxonomyStore`) et scraper (`CategorizationService`)
ne font que des SELECT ; la WebConsole ne touche aucune des trois tables.
Activation **non bloquante**.

## Lot 2 — PROPOSÉ, NON APPLIQUÉ (`profiles`) — **P0**

Voir `20260605_lot2_profiles_rls_PROPOSAL.sql`.
- État actuel : RLS off → lecture/écriture totale par anon (escalade `role`,
  contournement `status`/`ban_until`). C'est le risque le plus grave.
- Chemin de création confirmé : insert **client** (`authenticated`) dans
  `auth-provider.ts > ensureProfile()` avec `id = auth.uid()`,
  `role='particulier'`, `status='active'`. Pas de trigger `handle_new_user`.
- Bloqueur de validation : modèle de rôles (qui peut s'auto-attribuer quel rôle,
  comment se font les passages en `professionnel`/`institutionnel`).

## Note connexe (non sécurité, à vérifier fonctionnellement)

Plusieurs tables ont **RLS activé mais 0 policy** → totalement inaccessibles
aux clients anon/auth (lecture comprise), accès seulement via service_role /
fonctions `security definer` :
`active_boosts, activity_log, badges, contest_entries, contest_rewards,
contest_votes, contests, kv_store_b32ec4be, leaderboard, leaderboard_archive,
leaderboard_weekly, lumo_rules, lumo_transactions, missions, shop_items,
user_badges, user_inventory, user_levels, user_missions, user_preferences,
user_xp, xp, xp_rules`.

Ce n'est pas une faille (deny-by-default), mais si certaines de ces tables sont
censées être lues directement côté client (ex. `badges`, `shop_items`,
`leaderboard`, `user_preferences`), elles sont actuellement inaccessibles et
dépendent d'un accès serveur. À confirmer selon les besoins produit.
